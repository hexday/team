# 📂 کار با مخزن (Repository) – راهنمای عملی تیم

این ریپازیتوری قلب مستندات و کدهامونه. برای اینکه همه بتونیم راحت باهاش کار کنیم، به چندتا قانون ساده ولی مهم نیاز داریم.  
اگر با گیت آشنا نیستی نگران نباش – این راهنما رو قدم به قدم برو، خیلی زود دستت میاد.

```

## 🌿 شاخه‌ها (Branches) – ساده و کارآمد

برای اینکه کارها از روی هم نیفته، فقط از این سه نوع شاخه استفاده می‌کنیم:

| شاخه | توضیح | چه کسی می‌تواند مستقیم پوش (push) کند؟ |
|-------|--------|----------------------------------------|
| `main` | فقط کد پایدار و آماده تحویل | هیچکس (فقط از طریق Pull Request) |
| `develop` | شاخه اصلی کار روزانه | همه (با رعایت قوانین) |
| `feature/xyz` | برای هر تسک یا باگ جدید (مثلاً `feature/add-login`) | سازنده شاخه |

> **نکته:** هیچوقت مستقیم روی `main` کامیت نکن. همیشه ابتدا تغییرات را به `develop` یا یک شاخه feature بزن.

## 🔄 قدم‌های روزمره (چرخه کاری)

### ۱. دریافت آخرین تغییرات

```bash
git checkout develop
git pull origin develop
```

> این کار رو هر روز قبل از شروع کار انجام بده تا با بقیه هماهنگ باشی.

### ۲. ساختن شاخه جدید برای یک کار مشخص

```bash
git checkout -b feature/short-description
```

مثال: `feature/fix-api-login` یا `feature/add-report-template`

### ۳. کار کردن و کامیت زدن

تغییرات رو انجام بده، بعد:

```bash
git add .
git commit -m "پیام مفید و کوتاه"
```

**اصول پیام کامیت:**
- به فارسی یا انگلیسی (توی تیم مهم نیست، فقط یکدست باشه).  
- شروع با فعل: `add`، `fix`، `update`، `remove`.  
- مثال:  
  - `add weekly report template`  
  - `fix bug in user registration`  
  - `update roadmap-front.md`

### ۴. فرستادن شاخه به گیت‌هاب

```bash
git push origin feature/short-description
```

### ۵. ساختن Pull Request (PR)

- به گیت‌هاب برو، روی دکمه **Compare & pull request** کلیک کن.  
- عنوان و توضیح کوتاهی بنویس.  
- به عنوان reviewer حداقل یکی از ارشدها (مهدی / رضا / ارین) رو انتخاب کن.  
- بعد از تأیید، PR رو merge کن (معمولاً خودت انجام بده، مگر اینکه خواسته باشی دیگری انجام بده).

### ۶. حذف شاخه اضافی

بعد از merge شدن PR، توی گیت‌هاب گزینه‌ی **Delete branch** رو بزن. (اختیاری اما باعث نظم می‌شه.)

---

## 🧪 کار با شاخه `develop`

- این جاییه که همه اعضا به طور روزانه تغییراتشون رو میذارن.  
- قبل از push به `develop` حتماً این کارها رو بکن:  
  1. `git pull origin develop` (آخرین تغییرات رو بگیر)  
  2. اگر conflict (تداخل) داشت، حلش کن (راهنمای conflict توی فایل جدا هست)  
  3. بعد `git push origin develop`

> اگه حس کردی ممکنه تغییرت خراب کنه، بهتره اول یه feature branch بزنی و بعد PR بدی به `develop`.

---

## 🚨 قوانین مهم

- **هیچکس بدون Pull Request به `main` پوش (push) نمی‌کند.**  
- **قبل از شروع کار حتماً `git pull` بزن.**  
- **در پیام کامیت دروغ ننویس** (مثلاً «رفع باگ» در حالی که فقط یه فایل اضافه کردی).  
- **فایل‌های موقتی (`.pyc`, `__pycache__`, `.DS_Store`) رو هرگز اضافه نکن.** (اگه دیدی اشتباهی اضافه شدن، قبل از کامیت حذفشون کن.)

---

## 🆘 اگه اشتباه کردم یا تداخل (Conflict) دارم

اول نترس. دوم:

```bash
git status            # ببین کدوم فایل‌ها مشکل دارند
git diff              # اختلاف‌ها رو ببین
```

سپس فایل مورد نظر رو باز کن، خط‌هایی که با `<<<<<<<` و `>>>>>>>` مشخص شده رو ببین و تصمیم بگیر کدوم نسخه رو نگه داری. بعد از اصلاح:

```bash
git add .
git commit -m "resolve conflict in filename"
git push
```

اگر قاطی کردی و خواستی برگردی به حالت قبل:

```bash
git merge --abort
```

و بعد از یه نفر کمک بگیر.

- [ ] `git pull` زدی؟

- [ ] فایل‌های اضافی (cache, local config) رو stage نکردی؟  
- [ ] پیام کامیت واضح و مفید نوشتی؟  
- [ ] اگه feature branch بود، PR رو توی گیت‌هاب ساختی و یه نفر review کرد؟
